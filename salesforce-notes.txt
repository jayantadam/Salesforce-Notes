Data types

Text: Stores alphanumeric characters up to a specified length.

  Stores a single value chosen from a predefined list of options. Useful for enforcing data consistency (e.g., Status = Open, Closed, Pending).

  compared to single picklists.
Master-Detail Relationship:
  Creates a tight parent-child relationship between two objects where the child (detail) record is dependent on the parent (master) record. The parent controls record
  ownership, sharing, and deletion (if the master is deleted, the detail is also deleted). It supports roll-up summary fields on the master object to aggregate data 
  from child recordsFormula: Calculates values dynamically based on other fields.

Email: Stores email addresses.
Phone: Stores phone numbers.
URL: Stores website addresses.
Lookup Relationship: Links to another record in Salesforce.
Auto Number: Generates a unique sequential number automatically.
Long Text Area: Stores large blocks of text (up to 131,072 characters).
Rich Text Area: Stores formatted text with images and links.

  Supports dependent picklists for dynamic filtering.
Use Case: When you need controlled values for a field, not linking objects.

‚úÖ 1. Lookup Relationship
  Description: A loose association between two objects.
  Key Points:
  Child can exist without a parent.
  No cascade delete.
  No roll-up summary fields.
  Example: Contact linked to Account (but can exist independently).

‚úÖ 2. Master-Detail Relationship
  Description: A strong parent-child relationship.
  Key Points:
  Child record cannot exist without parent.
  Parent controls ownership, sharing, and deletion.
  Supports roll-up summary fields on the master.
  Example: Invoice Line Items tied to Invoice.

‚úÖ 3. Many-to-Many Relationship
  Description: Achieved using a junction object with two master-detail relationships. we can reduce duplicate records with help of this
  Key Points:
  Allows linking multiple records from two objects.
  Example: Students and Courses.

Delete Behavior
junction records can delete from both the parents Student and Events
    Because the junction object uses Master-Detail relationships, its records are dependent on both parents.
    If either parent record is deleted, all related junction records are automatically deleted (cascade delete).
    The other parent remains intact, but loses the link because the junction record is gone.

Example :- lets assume we have 2 parents Student and Event object and junction object (Student event info)

      1. If John (Student) is deleted:
        All his related junction records (registrations) are deleted.
        Events remain intact.

      2. If Salesforce Workshop (Event) is deleted:
         All junction records for that event are deleted.
         Students remain intact.

Security control :- can have first have first parernt object (in this cases student object bacuase we have event lookup)

‚úÖ 4. Hierarchical Relationship
  Description: Special lookup relationship available only on User object.
  Key Points:
  Used to create hierarchy among users (e.g., Manager ‚Üí Employee).

‚úÖ 5. External Lookup & Indirect Lookup
  Description: Used to link external objects (from external systems) to Salesforce objects.
  Key Points:
  External Lookup links external object to standard/custom object.
  Indirect Lookup links external object to Salesforce object using an external ID.

    a standard Boolean field called CascadeDelete, and its default value is true.

        Normally, deleting an Account does NOT auto-delete Opportunities

        You choose ‚ÄúDelete Account and Related Records‚Äù
        A Flow or Apex Trigger is configured
        Cascade Delete is enabled in Account Settings

lookup relationship - create a relationship to connects two objects loosely, allowing a child record to point to a parent record
 (like an Account to a Contact, or a custom object to a User) for data sharing, but the child can exist independently.
 it's defined on the child object by creating a new "Lookup Relationship" field, linking to the parent object, enabling easy data association without strict dependency or automatic deletion (unlike Master-Detail). 

Key Characteristics:
Loose Coupling: Child records aren't deleted if the parent is deleted (unless configured otherwise).
Optional: The lookup field on the child record doesn't have to be filled, allowing the child to exist without a parent.
One-to-Many: Typically links many child records to one parent record (e.g., many Contacts to one Account).
No Roll-Up Summaries: Cannot use Roll-Up Summary fields, which are a Master-Detail feature.
Creation: Created on the child object by selecting "Lookup Relationship" as the field type and choosing the parent object.

In Salesforce, a Profile is a fundamental security and access control component. It defines what a user can do in the Salesforce org.
1. Object Permissions
Determines what actions a user can perform on objects: Create,Read,Edit,Delete,View All,Modify All
Example: A Sales Rep may edit Opportunities but not modify Users.

2. Field-Level Security (FLS)
Specifies which fields a user can: See,Edit
Example: Salary field may be hidden for non-HR users.

3. User Permissions
Allows specific capabilities like: API Enabled,View Setup,Run Reports,Manage Dashboards

4. Tab Settings
Controls whether a tab is: Default On, Default Off,Hidden

5. Apps & Page Layout Assignments
Decides which: Apps users can access, Page layouts they see

6. Login Controls
Manages:Login IP ranges,Login hours

What Are Permission Sets in Salesforce?
    A Permission Set is a collection of settings and permissions that give users additional access on top of their Profile.
    Think of it as:
    üëâ Profile = Base access
    üëâ Permission Set = Extra access
    You use Permission Sets when you want to give someone more permissions without changing their profile.
How Many Permission Sets Can a User Have?
        A user can have:
        üëâ One Profile
        üëâ Many Permission Sets (no strict limit)

  üèÜ What Permission Sets Can Control
         ‚úîObject permissions,(Create, Read, Edit, Delete),‚úî Field‚Äëlevel security,(Make fields visible/editable),‚úî App permissions
      (Access to apps, tabs, features),‚úî System permissions,(API Enabled, View Setup, Modify All Data, etc.),‚úî User permissions,(Manage Reports, Run Flow, etc.)
        ‚úîRecord types,‚úî Apex class access,‚úî VF page access,‚úî Connected apps

Permission Sets vs Permission Set Groups
    Permission Set Groups allow you to combine multiple permission sets into one bundle.
You use them when:
    You have many permission sets
    You want to assign them together efficiently
    You want to remove overlapping permissions using Muting
